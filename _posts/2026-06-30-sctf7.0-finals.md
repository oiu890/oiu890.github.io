## haven
### Challenge protections
![haven](/assets/images/sctf7.0/haven.png)  
No PIE  
Source code:  
```c
// gcc -o chal chal.c -Wl,-z,relro,-z,now -fno-pie -no-pie
#include <stdio.h>
#include <stdint.h>
#include <unistd.h>

void vuln() {
    char buf[8]={0};
    uint64_t exit=0;
    read(0, buf, 32);    
    if (exit == -1) return;
    vuln();
    return;
}

int main() {
    setbuf(stdin, NULL);
    setbuf(stdout, NULL);
    vuln();
    _exit(1295);
}
```
Relatively short and simple source. We have a 32 byte read into an 8 byte buffer, giving us an overflow. We can immediately think of some kind of ROP.

So it look like we are only able to control the 8 byte buffer, the exit variable, saved rbp and return address.... OR ARE WE? 🤔🤔🤔🤔  
Notice that `vuln()` is called recursively until `exit` is -1.  
```bash
pwndbg> x/20gx 0x7fffffffd890
```

Also notice that whenever `vuln()` is called, the stack frames are stacked on top of each other.  
Thus, if we recursively call `vuln()`, we can fill the stack with a ROP chain!  
I got this idea from a previous challenge I did [here](https://oiu890.github.io/posts/cyberthon25/#deceitful-treasure) from Cyberthon 2025, though that one only gave you a 1 null byte overwrite of saved RBP.  

#### ROP
Of so now we know how to ROP (you can test that it works by placing random gadgets such as `ret`), but what exactly should we do?  
Well...  
```bash
ROPgadget --binary ./chal_patched
...
0x00000000004011c1 : syscall
...
```
The binary very kindly gives us a syscall, through `_exit(1295)`. Seeing this, SROP comes to mind.  
However, for SROP, we need to somehow set our `RAX` register to be `15` (for `syscall 15`, `rt_sigreturn`).  
Unfortunately, we do not have any good gadgets for this, so we will have to do it a different way.  

#### Using read
One way that we can do this is by using the `read` call.  
`read` stores the number of bytes read into the `rax` register, so if we call read and enter exactly 15 bytes, we will have our `rax` control.  
(sorry im a bit too lazy to include proof and pictures)  
#### SROP
Thus, our workflow will be:  
1. place sigreturn frame on the stack through recursive calls  
    - our first frame will be to pivot to bss  
    - it should call `read` again to set up another chain, this time to call our shell  
2. place our `read` and `syscall` calls 
3. break out of the recursive call, our rop chain now executes  
4. send 15 bytes to trigger the first frame
5. we are now reading into `bss`
6. place another rop chain to call `read` again, as well as place our 2nd sigreturn frame  
7. read 15 bytes for syscall
   - this part required a bit more visualisation as you will ovewrite your own rop chain, as such, it is in this read where our `syscall` needs to be placed  
   - you can use gdb with breakpoints/pause to see where exactly your `rsp`/execution is before/during/after reading
8. `syscall` executes, we win
